# 컴포넌트 요청 가이드

제품 프로젝트에서 와이어프레임을 그린 뒤, 필요한 UI 요소가 디자인 시스템에 있는지 대조하고
없는 것을 요청하는 절차다. [컴포넌트 인벤토리](./COMPONENTS.md)가 대조 기준이다.

## 먼저: 대부분은 "없는 것"이 아니다

와이어프레임의 요소를 인벤토리와 대조하면 "DS에 없다"가 잔뜩 나온다. 그중 실제로 DS를
고쳐야 하는 건 절반이 안 된다. 각 요소를 네 갈래로 나눈다.

| 유형 | 무엇인가 | 어디서 처리하나 |
|---|---|---|
| **a** | 그대로 있다 | 끝. 그냥 쓴다 |
| **b** | 컴포넌트는 있는데 필요한 변형·prop이 없다 | webui 이슈 (확장) |
| **c** | 기존 컴포넌트들의 **조합**이다 | 제품 repo. 이슈 없음 |
| **d** | 진짜 새 원시 요소다 | 제품 repo에서 먼저 만들고 → webui 이슈 |

**c가 가장 많고, c를 DS로 올리는 것이 디자인 시스템이 망가지는 가장 흔한 경로다.**
"필터 바"는 `SearchInput` + `Select` + `Chip`이다. "사용자 카드"는 `Card` + `Avatar`… 다.
제품 화면의 조합물은 제품이 소유한다.

b와 d를 가르는 질문: **이걸 기존 컴포넌트의 prop 하나로 표현할 수 있나?**
`Chip`에 `dismissible`을 더하면 되는 일이라면 b다. `DatePicker`는 어떤 prop으로도 안 되니 d다.

## d에는 문턱이 있다 — 세 곳 규칙

한 화면에서만 쓰는 것은 DS에 올리지 않는다. **세 번째 사용처가 생겼을 때** 승격한다.
그전까지는 제품 repo의 `src/ds-candidates/`에 둔다.

이 순서를 지키면 제품이 DS를 기다리며 멈추는 일이 없다. 그리고 승격 시점에는 이미
실전에서 API가 다듬어져 있어서, 이슈에 "만들어 주세요" 대신 "이렇게 돌아가고 있으니
검토해 주세요"를 쓸 수 있다. 스펙 작성 비용이 거의 0이 된다.

## 어느 repo에 내는가

- **컴포넌트·변형·prop이 없다** → 이 repo(`design-system-v2-webui`) 이슈
- **값이 없다** (색상·간격·라운드·타이포 토큰) → 파운데이션 repo(`design-system-v2`) 이슈

파운데이션은 Stage 1이라 색상 토큰만 있다. 간격이나 라운드 값이 없어서 막히는 일이
실제로 생긴다. 그건 여기가 아니라 파운데이션에 낸다. webui 이슈가 파운데이션 토큰에
막혀 있다면 `blocked-on-foundation` 라벨을 달고 파운데이션 이슈를 링크한다.

## ⚠️ 이 repo는 공개다

이슈도 공개된다. 제품 와이어프레임 캡처, 미공개 제품 정보, 내부 화면명·고객사명을
붙이지 않는다.

갭 이슈는 제품의 언어가 아니라 **컴포넌트의 언어**로 쓴다. "OnMyFactory 설비 상세 화면의
가동률 추이"가 아니라 "시계열 차트 위에 얹는 기간 선택기". 어차피 그게 좋은 DS 이슈다 —
한 제품에 묶이지 않아야 재사용 가능한 컴포넌트가 된다.

## 절차

1. 와이어프레임을 그리는 자리에서 제품 repo의 `docs/ds-gap.md`에 요소를 적는다.
   화면별 섹션, 각 행은 `요소 / 유형(a·b·c·d) / DS 컴포넌트 또는 갭 이름`.
   나중에 코드 짤 때 하는 것보다 정확하다.
2. 전체 화면을 다 매핑한다.
3. **한 번에 몰아서** 이슈를 낸다. 컴포넌트당 이슈 하나, 본문에 사용처 화면들을 나열한다.
   화면 단위로 찔끔 내면 "5개 화면이 전부 `DatePicker`를 원한다"는 신호가 다섯 이슈에
   흩어져 보이지 않는다. 그 신호가 티어와 우선순위를 정한다.
4. d로 분류된 것은 `src/ds-candidates/`에 만들어 제품을 먼저 출시하고, 이슈에 구현을 링크한다.

이슈는 [컴포넌트 요청 템플릿](https://github.com/CREFLEINC/design-system-v2-webui/issues/new/choose)으로 낸다.
