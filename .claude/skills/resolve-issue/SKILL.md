# Resolve Issue — GitHub 이슈 해결 팀 오케스트레이터

GitHub 이슈를 분석 → 구현 → 검증 → PR 생성으로 이어지는 파이프라인으로 해결한다.

## 실행 모드: 서브 에이전트 (감독자 패턴)

메인 세션(당신)은 **디스패처**다. 의사결정은 분석가가 하고, 당신은 이슈 조달·에이전트 스폰·보고 중계·git/PR 작업만 기계적으로 수행한다. 분석가의 판단을 재해석하거나 덮어쓰지 않는다 — 역할이 섞이면 판정 근거가 무너진다.

한 번에 이슈 하나를 처리한다. 여러 이슈를 요청받으면 이슈별로 이 파이프라인을 순차 반복한다(브랜치·PR 분리).

## 에이전트 구성

| 에이전트 | subagent_type | model | 역할 | 출력 |
|---------|--------------|-------|------|------|
| 분석가 | `issue-analyst` | `fable` | 요구사항 분석, 설계, 태스크 분해·등급 배정, 검증 계획, 보고 판정 | `01_analysis.md` |
| 고급 실무자 | `issue-worker-senior` | `opus` | 난이도 7점 이상 태스크 구현 + 자가 검증 | `02_task-{ID}_report.md` |
| 일반 실무자 | `issue-worker-junior` | `sonnet` | 난이도 7점 미만 태스크 구현 + 자가 검증 | `02_task-{ID}_report.md` |
| 검증자 | `issue-tester` | `sonnet` | 검증 계획 그대로 실행, 무편집 보고 | `03_test_report.md` |

- Agent 호출 시 항상 `model` 파라미터를 위 표대로 명시한다.
- `fable` 모델을 사용할 수 없는 환경이면 분석가는 `opus`로 대체한다.
- 분석가는 **이슈당 1명, 파이프라인 내내 유지**한다: 최초 1회 Agent로 스폰한 뒤, 이후 모든 판정 요청은 같은 에이전트에 `SendMessage`로 보낸다(컨텍스트 유지). 새로 스폰하면 판정 맥락이 사라진다.

## 데이터 전달 프로토콜

- **파일 기반(산출물)**: 워크스페이스는 `_workspace/issue-{N}/`. 파일명 컨벤션은 `references/report-formats.md`의 형식을 따른다. 중간 산출물은 삭제하지 않는다(감사 추적용).
- **반환값/SendMessage(조율)**: SendMessage로 요청을 전달한다. 단, 에이전트 반환 메시지는 유실될 수 있으므로(운영에서 실측된 문제) 완료 신호로만 취급한다 — 내용 있는 산출물·판정은 반드시 파일로 받는다.
- **판정 파일(분석가)**: 모든 판정은 파일로 기록받는다 — wave 판정 `04_wave{i}_verdict.md`, 최종 판정 `04_final_verdict.md`(PASS 시 PR 본문 초안 포함). 판정 요청 시 이 경로를 명시하고, 완료 알림 후 파일을 읽어 진행한다.
- 에이전트에게 전달하는 프롬프트에는 항상: 태스크 ID(해당 시), 워크스페이스 절대 경로, 읽어야 할 파일 경로, 출력 파일 경로를 명시한다.

## 워크플로우

### Phase 0: 컨텍스트 확인 (후속 작업 지원)

`_workspace/issue-{N}/` 존재 여부로 실행 모드를 결정한다:

- **미존재** → 초기 실행. Phase 1로 진행.
- **존재 + 부분 재실행 요청**(예: "검증만 다시", "PR 리뷰 반영") → 해당 Phase만 재실행. 분석가를 새로 스폰하되 "기존 `01_analysis.md`를 읽고 갱신하라"고 지시한다.
- **존재 + 같은 이슈를 처음부터 다시** → 기존 워크스페이스를 `_workspace/issue-{N}_prev_{YYYYMMDD}/`로 이동 후 초기 실행.

### Phase 1: 준비

1. 이슈 조달: `gh issue view {N} --json number,title,body,labels,comments` → `_workspace/issue-{N}/00_issue.md`에 제목·본문·라벨·코멘트를 저장.
2. `.gitignore`에 `_workspace/`가 있는지 확인, 없으면 추가.
3. 브랜치 생성: 최신 main 기준 `issue/{N}-{짧은-슬러그}`. 이미 있으면 체크아웃 후 재사용.

### Phase 2: 분석

1. Agent 호출: `subagent_type: "issue-analyst"`, `model: "fable"`, `run_in_background: false`.
   - 프롬프트에 포함: 워크스페이스 경로, `00_issue.md` 경로, "분석 계획서를 `01_analysis.md`로 작성하고 디스패치 순서를 반환하라".
2. 반환된 디스패치 계획(wave별 태스크·등급)을 확인한다. 반환 메시지가 비어 있거나 불충분하면 `01_analysis.md`의 태스크 섹션에서 wave·등급·대상 파일을 직접 읽는다(계획서가 단일 진실 원천). `BLOCKED` 반환 시 사유를 사용자에게 보고하고 종료.
3. 분석가의 에이전트 ID/이름을 기억해 둔다 — 이후 판정은 전부 SendMessage.

### Phase 3: 구현 (wave 단위 반복)

각 wave에 대해:

1. wave 내 태스크들의 실무자를 **한 메시지에서 동시에** Agent 호출한다(병렬). 등급에 따라 `issue-worker-senior`(model `opus`) / `issue-worker-junior`(model `sonnet`).
2. 전원 완료 대기 → 반환값 수집.
3. 분석가에게 SendMessage: "wave-{i} 보고서(`02_task-*_report.md`)를 검토하고 태스크별 승인/재작업을 판정해 `04_wave{i}_verdict.md`에 기록하라". 완료 알림 후 판정 파일을 읽어 진행한다.
4. 재작업 판정 시: 해당 실무자에게 SendMessage로 수정 지시를 전달(같은 실무자 재사용, 컨텍스트 유지). 태스크당 최대 2회. 초과 시 사용자에게 보고하고 지시를 기다린다.
5. `BLOCKED` 반환 시: 분석가에게 중계하여 계획 수정 또는 중단을 판정받는다.
6. 전 태스크 승인 후 다음 wave로.

### Phase 4: 검증

1. Agent 호출: `subagent_type: "issue-tester"`, `model: "sonnet"`. 프롬프트: "`01_analysis.md`의 검증 계획을 그대로 실행하고 `03_test_report.md`를 작성하라".
2. 분석가에게 SendMessage: "검증 보고서를 판정(PASS/FAIL)해 `04_final_verdict.md`에 기록하라 — PASS 시 PR 본문 초안(수용 기준 대비 충족 현황 포함)도 함께". 완료 알림 후 판정 파일을 읽어 진행한다.
3. `FAIL` → 분석가가 반환한 수정 태스크로 Phase 3 재진입(재검증 시 검증자는 `03_test_report_r{회차}.md` 작성). 검증 사이클 최대 2회. 초과 시 실패 내용 전체를 사용자에게 보고하고 중단.
4. `PASS` → Phase 5로.

### Phase 5: PR 생성

1. 변경 사항 커밋: 리포 커밋 컨벤션을 최근 `git log`에서 확인해 따른다(이 리포: `feat:`/`fix:` + 한국어 요약). 기본 1커밋, 분석가 계획이 논리 단위 분리를 명시했으면 그에 따른다.
2. push 후 `gh pr create` — 본문에 포함: 이슈 요약, 구현 내용(분석 계획서 기반), 검증 결과(검증 보고서 요약), `Closes #{N}`.
3. 사용자에게 최종 보고: PR URL, 태스크별 등급·재작업 이력, 검증 결과 요약, BLOCKED/미해결 항목.
4. 피드백 기회 제공: "결과나 팀 운영 방식에서 개선할 부분이 있으면 알려달라" 한 줄. 피드백은 하네스 진화(CLAUDE.md 변경 이력)에 반영한다.

## 에러 핸들링

| 상황 | 전략 |
|------|------|
| 분석가 BLOCKED | 사유를 사용자에게 보고하고 종료. 태스크 강행 금지 |
| 실무자 1명 실패/무응답 | 같은 등급으로 1회 재스폰. 재실패 시 분석가에게 중계해 계획 수정 판정 |
| 재작업 2회 초과 | 사용자에게 보고하고 지시 대기 |
| 검증 사이클 2회 초과 | 실패 출력 전문과 함께 사용자에게 보고하고 중단. PR 만들지 않음 |
| `gh` 인증/네트워크 실패 | 1회 재시도 후, 로컬 산출물(브랜치·커밋)까지 완료하고 PR 단계만 보류했음을 보고 |
| fable 모델 사용 불가 | 분석가를 `opus`로 대체하고 보고에 명시 |
| 태스크 간 파일 충돌 발견 | 병렬 중단, 해당 wave를 순차 실행으로 전환, 분석가에게 보고 |
| 에이전트 반환 메시지 유실/부실 | 파일 산출물(보고서·판정 파일)을 단일 진실 원천으로 읽는다. 기대한 파일이 없으면 SendMessage로 파일 기록을 재요청 |

## 테스트 시나리오

### 정상 흐름
1. 사용자: "이슈 16 해결해줘" → Phase 1에서 `00_issue.md` 생성, 브랜치 `issue/16-tabs-count-badge` 생성
2. 분석가가 태스크 3개 산출(예: T1 구현 6점 junior, T2 테스트 5점 junior, T3 스토리+문서 3점 junior; wave-1: [T1], wave-2: [T2, T3])
3. wave별 실무자 실행·승인 → 검증자 실행 → 분석가 PASS → 커밋·push·PR 생성
4. 예상 결과: `Closes #16` PR + `_workspace/issue-16/`에 보고서 전체 보존

### 에러 흐름
1. Phase 4에서 typecheck 실패 → 검증자가 에러 전문을 `03_test_report.md`에 기록
2. 분석가 FAIL 판정 + 수정 태스크 T4(junior) 반환 → Phase 3 재진입 → T4 완료·승인
3. 재검증(`03_test_report_r2.md`) → PASS → PR 생성, 최종 보고에 수정 이력 명시
4. 만약 2회 초과 실패 → PR 없이 실패 출력 전문을 사용자에게 보고
