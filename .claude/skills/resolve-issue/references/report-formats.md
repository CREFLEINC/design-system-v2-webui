# 보고서 형식 — resolve-issue 파이프라인 공통

모든 에이전트는 자기 산출물을 아래 형식으로 작성한다. 형식이 일정해야 다음 단계 에이전트가 기계적으로 읽을 수 있다.

워크스페이스: `_workspace/issue-{N}/`

| 파일 | 작성자 | 내용 |
|------|--------|------|
| `00_issue.md` | 오케스트레이터 | 이슈 원문 (제목/라벨/본문/코멘트) |
| `01_analysis.md` | issue-analyst | 분석 계획서 |
| `02_task-{ID}_report.md` | 실무자 | 실무자 보고서 (태스크당 1개) |
| `03_test_report.md` | issue-tester | 검증 보고서 (재검증 시 `_r{회차}` 접미) |
| `04_wave{i}_verdict.md` | issue-analyst | wave 판정 (태스크별 승인/재작업 + 근거) |
| `04_final_verdict.md` | issue-analyst | 최종 PASS/FAIL 판정 + (PASS 시) PR 본문 초안 |

> 에이전트 반환 메시지는 유실될 수 있다(운영 실측). **파일이 공식 채널**이며, 반환 메시지에는 한 줄 요약만 담는다.

---

## 분석 계획서 (`01_analysis.md`)

```markdown
# 이슈 #{N} 분석 계획서: {이슈 제목}

## 요구사항 요약
- {이슈에서 추출한 요구사항을 검증 가능한 문장으로}

## 가정
- {모호했던 부분과 채택한 해석. 없으면 "없음"}

## 구현 설계
{무엇을 어떻게 만드는지. 대상 파일, 공개 인터페이스(props/API), 변경 방식}

## 태스크

### T{번호}: {제목}
- **등급**: senior | junior | direct (합계 {n}/10 — 작업량 {a}/4, 외부 연계성 {b}/3, 복잡도 {c}/3)
- **wave**: {병렬 그룹 번호. 같은 wave = 동시 실행 = 대상 파일 겹침 금지}
- **의존**: 없음 | T{번호}
- **대상 파일**: {생성/수정할 파일 목록 — 병렬 충돌 판단 기준이므로 정확히}
- **지시 내용**: {실무자가 그대로 따라 할 구현 지시}
- **기대 결과**: {완료 시 보고해야 할 상태 — 검증 가능한 형태로}
- **자가 검증 방법**: {실무자가 실행할 명령어와 성공 기준}

(direct 태스크는 분석가가 처리 후 아래 한 줄 추가)
- **처리 결과**: {변경 파일과 결과}

## 검증 계획 (issue-tester 실행용)

| 단계 | 명령어 | 성공 기준 |
|------|--------|----------|
| 1 | {명령} | {exit 0, 특정 출력 포함 등} |
| ... | | |

## 판정 기준
- PASS 조건: {어떤 단계까지 성공하면 PASS인지}
- 알려진 허용 실패: {있다면 — 예: 기존에 깨져 있던 무관한 테스트}
```

## 실무자 보고서 (`02_task-{ID}_report.md`)

```markdown
# T{ID} 실무 보고서: {태스크 제목}

- **상태**: 완료 | BLOCKED
- **재작업 회차**: 0 | 1 | 2
- **변경 파일**: {목록}

## 수행 내용
{무엇을 어떻게 구현했는지 간결히}

## 자가 검증 결과
- 명령: `{실행한 명령}`
- exit code: {n}
- 출력 요약: {핵심 출력. 실패 시 에러 전문}

## 설계 이탈
{설계와 다르게 한 부분과 사유. 없으면 "없음"}

## 특이사항
{범위 밖에서 발견한 문제 등. 없으면 "없음"}
```

---

## 검증 보고서 (`03_test_report.md`)

```markdown
# 이슈 #{N} 검증 보고서 (회차 {r})

- **집계**: 성공 {n} / 실패 {m} / 실행불가 {k}

## 단계별 결과

### 단계 {i}: `{명령어}`
- 결과: 성공 | 실패 | 실행불가
- exit code: {n}
- 출력: {성공 시 핵심 요약 / 실패 시 에러 전문 — 잘라내지 않는다}

## 절차에 대한 의견
{절차 자체의 문제·개선 제안. 실행 결과에는 영향 없음. 없으면 "없음"}
```

## 판정 파일 (`04_wave{i}_verdict.md` / `04_final_verdict.md`)

```markdown
# {Wave-{i} | 최종} 판정 (issue-analyst, {날짜})

## {태스크별 승인/재작업 | PASS/FAIL}
{태스크별: "T{ID}: 승인 | 재작업(사유 + 구체적 수정 지시)" / 최종: "PASS | FAIL(수정 태스크 목록)"}

### 판정 근거
{보고서·diff·재실행에서 확인한 사실. 주장 재검증(스팟 체크) 결과 포함}

## 다음 단계
{오케스트레이터가 이어서 할 일 — 다음 wave 디스패치 / 재작업 지시 / 검증 디스패치 / PR 진행}

(최종 판정이 PASS인 경우) ## PR 본문 (오케스트레이터 사용용)
{제목 제안 + Closes #{N} + 요약 + 수용 기준 대비 충족 현황 체크리스트 + 설계 결정 + 변경 파일 + 검증 결과}
```
